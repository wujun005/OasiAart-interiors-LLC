# OasiAart Interiors LLC 项目宪章

**版本**: 1.0.0  
**生效日期**: 2024-12-19  
**最后修订日期**: 2024-12-19

## 项目概述

OasiAart Interiors LLC 是一个基于 Vue 3 的现代化 Web 应用程序，采用客户端和管理端分离的架构设计。项目使用 TypeScript 确保类型安全，通过 Vite 构建工具实现快速开发和构建。

### 技术栈

- **前端框架**: Vue 3 (Composition API)
- **开发语言**: TypeScript
- **构建工具**: Vite
- **状态管理**: Pinia
- **路由管理**: Vue Router
- **国际化**: Vue i18n
- **UI 组件库**: Ant Design Vue, Element Plus
- **样式预处理**: Sass/SCSS

## 核心原则

### 原则 1: 类型安全优先

**规则**:
- 所有新代码必须使用 TypeScript 编写，禁止使用 JavaScript。
- 禁用 `any` 类型，必须显式定义类型或使用泛型。
- 所有公共 API（组件 props、函数参数、返回值）必须有完整的类型定义。
- 使用 TypeScript 严格模式进行编译检查。

**理由**:
类型安全可以在编译时发现潜在错误，提高代码质量和可维护性，减少运行时错误。

### 原则 2: 模块化架构

**规则**:
- 代码必须按功能模块组织，客户端（client）和管理端（admin）完全分离。
- 每个模块包含独立的路由、状态管理、工具函数和样式。
- 共享代码放置在 `src/utils` 或 `src/components` 目录。
- 模块间通信通过明确的 API 接口或事件系统。

**理由**:
模块化设计提高代码可维护性，便于团队协作，支持功能独立开发和测试。

### 原则 3: 组件化开发

**规则**:
- 使用 Vue 3 Composition API 编写组件，优先使用 `<script setup>` 语法。
- 组件必须单一职责，保持小而专注。
- 组件 Props 必须有类型定义和默认值（如适用）。
- 可复用组件放置在共享目录，业务组件放置在对应模块目录。

**理由**:
组件化开发提高代码复用性，降低耦合度，便于测试和维护。

### 原则 4: 国际化支持

**规则**:
- 所有面向用户的文本内容必须使用 i18n 进行国际化处理。
- 禁止在代码中硬编码中文字符串。
- 语言文件按模块组织，保持中英文翻译同步。
- 使用有意义的翻译 key，遵循命名规范（如：`module.page.section.key`）。

**理由**:
国际化支持使应用能够服务多语言用户，提升用户体验，支持未来扩展。

### 原则 5: 路由管理规范

**规则**:
- 客户端路由以 `/index` 开头，管理端路由以 `/admin` 开头。
- 所有路由必须配置 meta 信息，包括 scope、requiresAuth、requiresRole 等。
- 路由守卫统一处理认证和授权逻辑。
- 路由懒加载必须使用动态 import。

**理由**:
清晰的路由规范便于维护和扩展，路由守卫统一处理安全逻辑，懒加载优化初始加载性能。

### 原则 6: 状态管理规范

**规则**:
- 使用 Pinia 进行状态管理，每个功能域创建独立的 store。
- Store 必须使用 TypeScript 定义状态类型。
- Store 的 actions 必须处理异步操作的错误情况。
- 避免在组件中直接修改 store 状态，必须通过 actions。

**理由**:
集中式状态管理使数据流清晰可控，便于调试和测试，支持状态持久化。

### 原则 7: API 请求规范

**规则**:
- 所有 API 请求必须使用封装的 request 工具函数。
- 请求必须包含统一的错误处理逻辑。
- API 响应必须定义 TypeScript 接口类型。
- 请求超时时间必须合理设置（默认不超过 30 秒）。

**理由**:
统一的请求封装便于添加认证、错误处理、日志记录等横切关注点，提高代码一致性。

### 原则 8: 代码风格一致性

**规则**:
- 遵循 Vue 3 官方风格指南。
- 使用 ESLint 和 Prettier 保持代码格式一致。
- 组件文件名使用 PascalCase，工具函数文件名使用 camelCase。
- 代码注释使用中文，关键业务逻辑必须有注释说明。

**理由**:
一致的代码风格降低团队协作成本，提高代码可读性，便于代码审查。

### 原则 9: 性能优化

**规则**:
- 路由必须使用懒加载，减少初始包大小。
- 图片资源必须优化，使用适当的格式和尺寸。
- 避免不必要的组件重新渲染，合理使用 computed 和 watch。
- 大型列表必须使用虚拟滚动或分页。

**理由**:
性能优化直接影响用户体验，合理的优化策略可以显著提升应用响应速度。

### 原则 10: 安全性

**规则**:
- 所有用户输入必须进行验证和清理。
- 敏感信息（如 token）不得存储在 localStorage，应使用更安全的存储方式。
- API 请求必须包含认证信息，管理端路由必须进行权限校验。
- 禁止在前端代码中硬编码 API 密钥或敏感配置。

**理由**:
安全性是 Web 应用的基本要求，必须从设计阶段就考虑安全措施，防范常见安全漏洞。

## 治理

### 修订程序

1. **提议修订**: 任何团队成员可以提议修订项目宪章。
2. **讨论和评审**: 提议需要在团队会议上讨论，评估对项目的影响。
3. **版本更新**: 修订通过后，必须更新版本号（遵循语义化版本规范）。
4. **文档同步**: 修订后必须同步更新相关的模板和文档。

### 版本控制

版本号遵循语义化版本规范（Semantic Versioning）:
- **主版本号 (MAJOR)**: 向后不兼容的原则变更或原则移除。
- **次版本号 (MINOR)**: 新增原则或现有原则的重大扩展。
- **修订版本号 (PATCH)**: 澄清说明、措辞修正、非语义性改进。

### 合规审查

- 新功能开发前必须审查是否符合项目宪章原则。
- 代码审查时必须检查是否违反宪章规则。
- 定期（建议每季度）审查宪章的有效性和适用性。
